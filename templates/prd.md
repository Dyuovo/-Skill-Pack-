# PRD

> 用途：把一个想法或需求碎片变成可判断、可验收、可拆任务的产品定义。  
> 写入位置：`docs/changes/REQ-YYYY-NNN/prd.md`

## Metadata

- Project:
- Change ID:
- Owner:
- Status: draft / review / accepted
- Updated:

## User

谁会使用或受影响？写具体角色，不写泛泛的“用户”。

## Problem

这个角色现在遇到什么问题？为什么值得解决？如果是系统改造，写当前架构缺口。

## Goal

本次变更要达成什么结果？最好能用可观察结果描述。

## Non-Goals

明确本次不做什么，防止范围膨胀。

## Acceptance Criteria

用 Given/When/Then 或可检查清单描述。每条必须能被测试或人工验收。

- Given ..., when ..., then ...

## Risks

列出产品、技术、数据、安全、运维或协作风险，并写缓解方式。

| Risk | Impact | Mitigation |
|---|---|---|
